<objective>
Implement the two concrete adapters that make Phase 1 actually run: a `HackerNewsSource` that fetches the HN front page via the public Firebase API (no auth), and a `SqlAlchemyTopicRepository` that implements the dedup-aware upsert flow against Postgres. After this plan, both adapters are individually tested — the source against a recorded HTTP response, the repository against a real Postgres test database.

`autonomous: false` — the repository tests need a live Postgres. Operator confirms a test DB is available before running pytest.
</objective>

<must_haves>
  <truths>
    - `HackerNewsSource.name == "hackernews"` (this string is persisted in `topic_sources.source_name`)
    - HN API: GET `https://hacker-news.firebaseio.com/v0/topstories.json` returns `list[int]` of story ids; GET `https://hacker-news.firebaseio.com/v0/item/{id}.json` returns the story object
    - `native_rank` is the index in the topstories array + 1 (1-based)
    - URL field: HN items have a `url` (external link) OR no url for self-posts (Ask HN, Show HN). For self-posts, fall back to `https://news.ycombinator.com/item?id={id}`
    - The repository's `find_candidates` query uses Postgres trigram index for cheap candidate retrieval (good enough; the actual dedup decision is made in the domain layer with rapidfuzz). For Phase 1 simplicity we can also start with a recent-window scan and add the trigram index later if performance demands it
    - `update_existing` MUST be idempotent — re-inserting the same `(topic_id, source_name, url, observed_at)` raises a unique-violation that is caught and ignored
    - ING-008: re-crawls bump `last_seen_at` and increment `observation_count` exactly once per crawl (not once per duplicate row attempted)
  </truths>
</must_haves>

  <action>
    Create `services/crawler/src/crawler/adapters/sources/hackernews.py` defining `class HackerNewsSource:` with:
    - `name = "hackernews"` (class attribute)
    - `__init__(self, http_client: httpx.AsyncClient | None = None, base_url: str = "https://hacker-news.firebaseio.com/v0")` — accept an optional `httpx.AsyncClient` to enable injection for tests; if None, create one internally with a 10s timeout
    - `async def fetch(self, top_n: int) -> list[RawItem]`:
      1. GET `{base_url}/topstories.json` → `list[int]` of story ids
      2. Slice to first `top_n` ids
      3. For each id `i` at 1-based index `rank`, GET `{base_url}/item/{i}.json` concurrently using `asyncio.gather` (limit concurrency to 10 with `asyncio.Semaphore(10)` to avoid hammering the API)
      4. For each item dict, build a `RawItem`:
         - `title = item["title"]`
         - `url = item.get("url") or f"https://news.ycombinator.com/item?id={item['id']}"`
         - `source_name = "hackernews"`
         - `native_rank = rank` (1-based index in original topstories array)
         - `observed_at = datetime.now(timezone.utc)`
         - `raw_payload = item` (the full dict)
      5. Skip items where `item` is None or `"title"` is missing (HN sometimes returns `null` for deleted stories) — log a warning via `structlog.get_logger().warning("hn.missing_item", id=i)` and continue
      6. Return the list of `RawItem` (preserving rank order)

    If `http_client` was created internally, close it in a `finally` block. Use `async with httpx.AsyncClient(...) as client` pattern when self-managed.

    Verify it conforms to `SourcePort` at runtime: at module bottom (or in a `__post_init__`-like location) the test will assert `isinstance(HackerNewsSource(), SourcePort)` works because `SourcePort` is `@runtime_checkable`.
  </action>

<task id="01-04-T03">
  <title>Implement SqlAlchemyTopicRepository adapter</title>
  <read_first>

  </read_first>
  <action>
    Create `services/crawler/src/crawler/adapters/persistence/sqlalchemy_topic_repository.py` defining `class SqlAlchemyTopicRepository:`:

    - `__init__(self, session_factory: async_sessionmaker[AsyncSession])` — store the factory (each method opens its own session, commits, closes)

    - `async def find_candidates(self, dedup_key: str, limit: int = 50) -> list[TopicCandidate]`:
      Execute SQL using `func.lower()` and `regexp_replace` to compute the same normalization on `topics.title` server-side, then use `LIKE` with `%` wildcards on the most distinctive token to get cheap candidates. For Phase 1, simpler approach: SELECT top `limit` topics ordered by `last_seen_at DESC` (recent-window scan) — this works because most active topics are recent. Future plans can swap to trigram-indexed search.
      Map each row to `TopicCandidate(id=row.id, title=row.title, last_seen_at=row.last_seen_at, observation_count=row.observation_count)`.

    - `async def insert_new(self, item: RawItem) -> UUID`:
      Open a session. Create a `Topic` with `title=item.title`, `description=None`, `topic_metadata={}` (or `{"first_source": item.source_name}`), `observation_count=1`. Add and flush to get the `id`. Create a `TopicSource` with `topic_id=topic.id`, `source_name=item.source_name`, `url=item.url`, `native_rank=item.native_rank`, `observed_at=item.observed_at`, `raw_payload=item.raw_payload`. Add, commit, return `topic.id`.

    - `async def update_existing(self, topic_id: UUID, item: RawItem) -> None`:
      Open a session. UPDATE topic: SET `last_seen_at = item.observed_at`, `observation_count = observation_count + 1`, `updated_at = now()` WHERE `id = topic_id`. Then INSERT a new TopicSource row. If the unique constraint `(topic_id, source_name, url, observed_at)` is violated (re-running the same crawl in the same second), catch `IntegrityError` and ROLLBACK the session — re-open and re-execute ONLY the topic UPDATE (so we still bump last_seen / observation_count even when the source row was a dup). On success, commit. (Note: the increment-once-per-crawl invariant from must_haves is achieved at the orchestrator level in plan 01-05 — this method just does what's asked atomically.)

    Use `from sqlalchemy import select, update, func` and `from sqlalchemy.exc import IntegrityError`. Use `from core.models import Topic, TopicSource`.
  </action>
