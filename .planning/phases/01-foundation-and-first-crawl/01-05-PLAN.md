<objective>
Wire the composition root, the Typer CLI (`crawler run-once`), the orchestrator that runs the dedup-and-upsert flow, and the `docker-compose.yml` that brings up Postgres + the crawler service. After this plan, `docker compose up postgres -d && uv run alembic upgrade head` (with DATABASE_URL set) prepares the DB, and `docker compose run --rm crawler run-once --top-n 30` (or its `uv run` equivalent) inserts ~30 deduplicated HN topics into Postgres and exits cleanly. Running it a second time does NOT create duplicates — existing topics are updated.

This plan is the integration vertical slice that makes Phase 1 satisfy success criteria 3, 4, and 5.

`autonomous: false` — needs Docker installed and operator to run commands.
</objective>

<must_haves>
  <truths>
    - The crawler is a stateless one-shot job — process exits 0 after a single crawl (locked architectural decision, no in-process scheduler)
    - Composition root is `services/crawler/src/crawler/app/composition.py` — it's the ONLY place that knows how to wire ports to adapters
    - The orchestrator (`run_once`) does: for each registered source → fetch RawItems → for each item, find candidates, run dedup decision in domain, decide insert vs update, persist
    - "Increment observation_count exactly once per crawl" invariant from 01-04 must_haves: the orchestrator deduplicates within a single crawl run using a Python `set` of `topic_id`s already updated this run, so the same topic seen by multiple sources in one run only bumps observation_count once
    - Logs are structured (structlog) with these fields per crawl: `source_name`, `top_n`, `fetched`, `inserted`, `updated`, `skipped_duplicates_within_run`, `errors`, `duration_ms`
    - Docker-compose runs the crawler container with the workspace .venv mounted? No — image bakes deps via `uv sync --frozen`
    - ING-003: HN top-N defaults to 100 but is overridable via `--top-n` CLI flag and `CRAWLER_TOP_N` env var (CLI wins)
  </truths>
</must_haves>

  <read_first>

  <action>
    Create `services/crawler/src/crawler/app/orchestrator.py` exposing `async def run_once(sources: list[SourcePort], repo: TopicRepositoryPort, top_n: int) -> dict[str, Any]`.

    Algorithm:
    1. Initialize `stats = {"sources": {}, "totals": {"fetched": 0, "inserted": 0, "updated": 0, "skipped_within_run": 0, "errors": 0}, "started_at": datetime.now(timezone.utc).isoformat()}`
    2. Initialize `updated_this_run: set[UUID] = set()` — tracks which topic_ids have already been bumped this run, to enforce the increment-once invariant
    3. For each `source` in `sources`:
       a. Try `items = await source.fetch(top_n)` — on exception, log error, increment errors counter, continue to next source
       b. For each `item` in `items`:
          - Compute `key = dedup_key(item.title)`
          - `candidates = await repo.find_candidates(key, limit=50)`
          - For each `candidate` in candidates: if `is_duplicate(item.title, candidate.title)` → match, break
          - If a match was found:
            - If `candidate.id in updated_this_run`: increment `skipped_within_run`, continue (don't bump observation_count again, but DO still want to record the source observation? For Phase 1 simplicity: skip the entire update for cross-source dups within one run. This is documented as "intentional Phase 1 simplification — see Phase 2 for cross-source aggregation refinement.")
            - Else: `await repo.update_existing(candidate.id, item)`, add to `updated_this_run`, increment `updated`
          - Else: `topic_id = await repo.insert_new(item)`, add to `updated_this_run`, increment `inserted`
       c. Record per-source stats
    4. Add `finished_at` and `duration_ms` to stats. Log a summary line via `structlog`. Return `stats`.

    The async wrapper that the CLI calls should run this with `asyncio.run(...)`.
  </action>
  <acceptance_criteria>
    - File `services/crawler/src/crawler/app/orchestrator.py` exists

    Create `services/crawler/src/crawler/app/cli.py` defining a `typer.Typer()` app with one command `run_once`:

    ```python
    import asyncio
    import os
    import typer
    from crawler.app.composition import build_sources, build_repository
    from crawler.app.logging import configure_logging
    from crawler.app.orchestrator import run_once as run_once_async

    app = typer.Typer(help="Trend Researcher crawler — one-shot job runner.")

    @app.command("run-once")
    def run_once_cmd(
        top_n: int = typer.Option(
            None, "--top-n", "-n",
            help="Items per source. Defaults to CRAWLER_TOP_N env var or 100."
        ),
        log_level: str = typer.Option("INFO", "--log-level", help="DEBUG/INFO/WARNING/ERROR"),
    ) -> None:
        """Run a single crawl across all configured sources, then exit."""
        configure_logging(level=log_level)
        effective_top_n = top_n if top_n is not None else int(os.getenv("CRAWLER_TOP_N", "100"))
        sources = build_sources()
        repo, engine = build_repository()
        try:
            stats = asyncio.run(run_once_async(sources, repo, effective_top_n))
            typer.echo(f"crawl complete: {stats['totals']}")
        finally:
            asyncio.run(engine.dispose())

    if __name__ == "__main__":
        app()
    ```

    FROM ghcr.io/astral-sh/uv:python3.12-bookworm-slim AS builder

    ENV UV_LINK_MODE=copy \
        UV_COMPILE_BYTECODE=1 \
        UV_PYTHON_DOWNLOADS=never

    WORKDIR /app

    # Copy the workspace root manifest + lockfile first (layer cache)
    COPY pyproject.toml uv.lock .python-version ./

    # Copy all workspace member manifests so uv can resolve the graph
    COPY packages/core/pyproject.toml packages/core/pyproject.toml
    COPY services/crawler/pyproject.toml services/crawler/pyproject.toml
    COPY services/api/pyproject.toml services/api/pyproject.toml
    COPY services/assessor/pyproject.toml services/assessor/pyproject.toml

    # Sync only the deps tree (no project source yet) for max cache reuse
    RUN --mount=type=cache,target=/root/.cache/uv \
        uv sync --frozen --no-install-project --package crawler

    # Now copy actual source for crawler + core (the only two needed at runtime)
    COPY packages/core packages/core
    COPY services/crawler services/crawler

    # Install the project itself
    RUN --mount=type=cache,target=/root/.cache/uv \
        uv sync --frozen --package crawler

    # Final runtime image
    FROM python:3.12-slim-bookworm AS runtime

    ENV PATH="/app/.venv/bin:$PATH" \
        PYTHONDONTWRITEBYTECODE=1 \
        PYTHONUNBUFFERED=1

    WORKDIR /app
    COPY --from=builder /app /app

    # Default to the CLI entrypoint
    ENTRYPOINT ["crawler"]
    CMD ["run-once"]

    9. `docker compose run --rm crawler run-once --top-n 30` (run AGAIN)

    11. `docker compose exec postgres psql -U trend -d trend_researcher -c "SELECT max(observation_count) FROM topics;"` — record (call it OC)
    12. `docker compose exec postgres psql -U trend -d trend_researcher -c "SELECT count(*) FROM topic_sources;"` — record (call it S2)

    If `N2 - N1 > 5`: dedup is not working as expected — investigate before passing the phase. Some increase is OK because HN's front page genuinely changes between crawls (new stories appear, old stories drop). Tolerance: `N2 - N1 <= 5` is acceptable.

    If `OC < 2` after the second crawl: `update_existing` isn't bumping `observation_count` correctly — investigate.

    If `S2 < S1`: something deleted source rows — bug. Should be `S2 >= S1` (new observation rows added on the second crawl, even for already-known topics, due to new `observed_at` timestamps).
