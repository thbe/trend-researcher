# Plan 03-04: Scheduler service + compose-wide wiring (Wave 4)

**Phase**: 3 â€” Scheduler & Ops Baseline
**Mode**: mvp
**Depends on**: Plan 03-02 (api needs to be in compose), Plan 03-03 (env-disable should already work â€” scheduler-triggered runs respect it)

## Goal
Add a `scheduler` compose service that fires `docker compose run --rm crawler run-once` every 12h, and wire `api` + `scheduler` into `docker-compose.yml`. After this plan, a single `docker compose up -d` brings up Postgres + api shell + scheduler, and the system runs unattended.

## Slice shape (vertical)
container image (alpine + crond) â†’ mounted host docker socket â†’ crontab â†’ entrypoint â†’ compose orchestration of all 4 services (postgres, crawler image build, api, scheduler).

## Threat model (per Â§5.55)
- **Docker socket mount** = effectively root on host. Documented + accepted in CONTEXT.md Â§A.1 as the cost of "compose-self-schedules". Single-operator internal tool, attack surface considered acceptable. Documented in `services/scheduler/README.md`.
- Crontab is read-only inside the container; no code paths construct shell commands from external input.
- The scheduler container does not connect to Postgres or hold any DB credentials; only the crawler containers it spawns do. Lateral compromise via the scheduler doesn't grant DB access directly (still grants docker-socket access, which is the dominant risk above).

## Locked design decisions
- **Base image**: `alpine:3.20` (small, current stable; pinning to minor avoids silent base churn).
- **Cron daemon**: BusyBox `crond -f -l 8` (foreground, log level 8 = info).
- **Crontab anchor**: `0 0,12 * * *` UTC (00:00 and 12:00 UTC daily, anchored â€” operator can predict exact tick times for verification, no clock-drift surprises). UTC because compose containers default to UTC.
- **Trigger command**: `cd /workspace && docker compose run --rm crawler run-once`. The `/workspace` is the repo root mounted read-only into the scheduler container so `docker compose` can find `docker-compose.yml`.
- **Logging**: cron writes stdout/stderr to `/proc/1/fd/1` (PID 1 â€” entrypoint) so `docker logs trend-scheduler` shows both cron's own log lines and each crawl's output. Each crawl's structured logs already land in Postgres via `crawl_runs` (Plan 03-01) and in the crawler container's docker logs.
- **No restart policy on crawler** (preserved from Phase 1 â€” explicit comment kept).
- **Restart policy on scheduler**: `unless-stopped` (its job is to be up 24/7).
- **Restart policy on api**: `unless-stopped` (same â€” operator should be able to `curl /healthz` whenever).
