# Plan 03-02: api service shell — `/healthz` + `/runs` (Wave 2)

**Phase**: 3 — Scheduler & Ops Baseline
**Mode**: mvp
**Depends on**: Plan 03-01 (`crawl_runs` table must exist before `/runs` can read it)
**REQ-IDs in scope**: OPS-001 (compose-up brings api shell up), supports OPS-002 (operational visibility into crawl_runs)

## Goal
Promote `services/api/` from empty scaffold to a real running FastAPI service with two endpoints. After this plan, `docker compose up -d api` starts a container that responds to `GET /healthz` (200 + `{"status":"ok","db":"reachable"}`) and `GET /runs` (last N `crawl_runs` rows newest-first as JSON). No business endpoints, no auth — Phase 4 owns the topic-read product API.

## Slice shape (vertical)
HTTP route → handler → repository read → DB → JSON response, plus container packaging.

## Threat model (per §5.55)
- New HTTP service binds to compose network. Bound to host on port `8000` for operator curl. Single-operator internal tool — no auth in v1 (per ARC-004 / OOS).
- `/runs` is a read-only operational endpoint exposing crawl-run statistics (no PII, no titles). Worst case if leaked: an attacker learns when the crawler ran and how many topics it ingested.
- `/runs?limit=` is parameterized server-side (clamped 1–100) — no SQL injection vector even if the input is malicious.
- DB connection string read from `DATABASE_URL` env var (existing pattern from crawler).

## Locked design decisions
- **Framework**: FastAPI + uvicorn (industry standard, structured response + Pydantic validation come for free).
- **`/runs` pagination**: `LIMIT 20` default, `?limit=<int>` query param clamped to `[1, 100]`. No offset/cursor in v1 — operator only ever needs "the last few runs".
- **Order**: `started_at DESC` (newest first; matches operator mental model "what happened most recently").
- **DB access**: Reuse `core.get_engine` / `core.get_sessionmaker` (same wiring crawler uses). New api-side repository module is acceptable (read-only); do NOT push read-side code into `services/crawler/`.
- **Container port**: 8000 (FastAPI default).
- **No alembic in api startup**: api assumes migrations have been applied (they're applied by the operator pre-up, same as today). Phase 3 doesn't change that.

## Pattern map
- FastAPI app structure: `services/api/src/api/main.py` exports `app = FastAPI(...)`; routes live in `services/api/src/api/routes/<feature>.py` and are wired via `app.include_router(...)` in `main.py`.
- Pydantic response models: `services/api/src/api/schemas.py` contains `RunResponse`, `RunsListResponse`, `HealthzResponse`.
- Dependency-injected DB session: `services/api/src/api/dependencies.py` has `get_session()` async generator yielding an `AsyncSession`, used via `Depends(get_session)`.
- Tests: `services/api/tests/test_*.py` use `httpx.AsyncClient` against `app` via `ASGITransport` — no live network, no real DB unless `TEST_DATABASE_URL` set (skip-gate pattern from crawler).
- Dockerfile: model on `services/crawler/Dockerfile` — multi-stage uv install, copy `packages/core` + `services/api`, entrypoint runs uvicorn.

### T03 — `/healthz` route
- **read_first**: `services/api/src/api/dependencies.py` (T02), `services/api/src/api/schemas.py` (T02).
- **action**: New file `services/api/src/api/routes/healthz.py` with `router = APIRouter()` and `@router.get("/healthz", response_model=HealthzResponse)` async function `healthz(session: AsyncSession = Depends(get_session))`. Body: `await session.execute(text("SELECT 1"))`; on success return `{"status":"ok","db":"reachable"}`. Wrap in try/except: on `OperationalError` or any DBAPIError, return `JSONResponse(status_code=503, content={"status":"degraded","db":"unreachable"})`.
- Create `services/api/src/api/routes/__init__.py` (empty).
- **acceptance_criteria**:
  - With Postgres up: `curl -s localhost:8000/healthz` returns `{"status":"ok","db":"reachable"}` and HTTP 200 (verified via test in T05).
  - With Postgres down: HTTP 503 with `{"status":"degraded","db":"unreachable"}`.
- **commit**: `feat(03-02-T03): /healthz route with DB ping`

### T04 — `/runs` route
- **read_first**: `packages/core/src/core/models.py` (CrawlRun added in 03-01-T01), `services/api/src/api/schemas.py`.
- **action**: New file `services/api/src/api/routes/runs.py` with `router = APIRouter()` and `@router.get("/runs", response_model=RunsListResponse)` async function `list_runs(limit: int = Query(20, ge=1, le=100), session: AsyncSession = Depends(get_session))`. Body: `select(CrawlRun).order_by(CrawlRun.started_at.desc()).limit(limit)`; return `RunsListResponse(runs=[RunResponse.model_validate(r, from_attributes=True) for r in rows], limit=limit)`. Add `model_config = ConfigDict(from_attributes=True)` on `RunResponse`.
- **acceptance_criteria**:
  - `curl -s "localhost:8000/runs?limit=5"` returns valid JSON with `runs` array of length ≤ 5 and `limit:5` (verified in T06).
  - `curl -s "localhost:8000/runs?limit=999"` returns HTTP 422 (FastAPI validation rejects > 100).
  - `curl -s "localhost:8000/runs?limit=0"` returns HTTP 422 (rejects < 1).
  - Returned rows are ordered newest-first by `started_at`.
- **commit**: `feat(03-02-T04): /runs route — last N crawl_runs newest-first`
