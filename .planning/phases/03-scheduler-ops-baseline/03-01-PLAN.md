# Plan 03-01: `crawl_runs` table + write path (Wave 1)

**Phase**: 3 — Scheduler & Ops Baseline
**Mode**: mvp
**Depends on**: nothing (Phase 2 complete)
**REQ-IDs in scope**: OPS-002

## Goal
Land the operational telemetry table and the write path that fills it. After this plan, every `docker compose run --rm crawler run-once` invocation produces exactly one new `crawl_runs` row with the full per-source breakdown the orchestrator already computes. No new container, no scheduler, no API surface yet — those come in subsequent plans.

## Slice shape (vertical)
schema → migration → repository port → adapter → composition wiring → orchestrator integration → tests at unit + integration levels.

## Threat model (per §5.55)
New table holds operational counts only — no PII, no credentials, no user content (titles already live in `topics`). Same DB, same connection string, same trust boundary as `topics`/`topic_sources`. No new attack surface.

## Locked design decisions
- **Write timing**: end of `run_once()` only — one row per crawl. Captures the complete stats the orchestrator returns. A "started" row would double the writes for no operator benefit at v1 scale.
- **`failed_sources`**: stored as Postgres `text[]` (not JSONB) so SQL can do `cardinality(failed_sources) > 0` filters on `/runs` later without unwrapping JSON.
- **`per_source`**: stored as `JSONB` (mirrors orchestrator's nested per-source dict shape; flexible if shape evolves).
- **Migration revision id**: `0002` (follows existing `0001` Phase 1 pattern).

## Pattern map (file analogs to model on)
- New SQLAlchemy model: extend `packages/core/src/core/models.py` next to `Topic` / `TopicSource`. Same `Base`, same `mapped_column` style, same UUID-as-string PK with `gen_random_uuid()` server default.
- New Alembic migration: copy structure of `packages/core/alembic/versions/0001_initial_topics_and_topic_sources.py` (revision/down_revision header, `op.create_table` + `op.create_index`, mirrored `downgrade`).
- New repository port: shape after `services/crawler/src/crawler/ports/topic_repository_port.py` (Protocol class, async methods).
- New adapter: shape after `services/crawler/src/crawler/adapters/persistence/sqlalchemy_topic_repository.py` (constructor takes `async_sessionmaker`, methods open `async with self._session_factory() as session`).
- Composition wiring: extend `build_repository()` in `services/crawler/src/crawler/app/composition.py` to also return the new repo (or add a `build_crawl_run_repository()`).
- Orchestrator integration: pass the new repo into `run_once()` in `services/crawler/src/crawler/app/orchestrator.py` and write at the bottom of the function (just before the existing `_log.info("crawl.complete", ...)`).

## Tasks

### T01 — `CrawlRun` SQLAlchemy model
- **read_first**: `packages/core/src/core/models.py` (existing `Topic` / `TopicSource` patterns), `.planning/phases/03-scheduler-ops-baseline/CONTEXT.md` §B (column list).
- **action**: Add a `CrawlRun(Base)` class to `packages/core/src/core/models.py`. `__tablename__ = "crawl_runs"`. Columns:
  - `id`: UUID PK, server_default `gen_random_uuid()`.
  - `started_at`: TIMESTAMP(timezone=True), nullable=False.
  - `finished_at`: TIMESTAMP(timezone=True), nullable=False.
  - `duration_ms`: Integer, nullable=False.
  - `top_n`: Integer, nullable=False.
  - `totals_fetched`: Integer, nullable=False, server_default text("0").
  - `totals_inserted`: Integer, nullable=False, server_default text("0").
  - `totals_updated`: Integer, nullable=False, server_default text("0").
  - `totals_skipped_within_run`: Integer, nullable=False, server_default text("0").
  - `totals_errors`: Integer, nullable=False, server_default text("0").
  - `per_source`: JSONB, nullable=False, server_default text("'{}'::jsonb").
  - `failed_sources`: `ARRAY(Text)` (postgres dialect), nullable=False, server_default text("'{}'"::text[]).
  - `created_at`: TIMESTAMP(timezone=True), nullable=False, server_default text("now()").
  - Index on `started_at` (descending reads are the common access pattern via `/runs`).
- Append `CrawlRun` to module `__all__`.
- **acceptance_criteria**:
  - `uv run --package core python -c "from core.models import CrawlRun; print(CrawlRun.__tablename__)"` prints `crawl_runs`.
  - `uv run --package core python -c "from core.models import CrawlRun; print([c.name for c in CrawlRun.__table__.columns])"` lists all 14 columns above.
  - No edits to `Topic` / `TopicSource`.
- **commit**: `feat(03-01-T01): add CrawlRun ORM model in packages/core`

### T06 — Orchestrator integration
- **read_first**: `services/crawler/src/crawler/app/orchestrator.py` (existing `stats` dict shape), the new port from T03.
- **action**: Change signature of `run_once` to `async def run_once(sources: list[SourcePort], repo: TopicRepositoryPort, crawl_run_repo: CrawlRunRepositoryPort, top_n: int) -> dict[str, Any]`. Just before the closing `_log.info("crawl.complete", ...)`, build a `CrawlRunRecord` from the existing `stats` (parse the ISO timestamps back into `datetime` via `datetime.fromisoformat`, copy `top_n`, copy each `totals.<x>` to its `totals_<x>` field, copy `stats["sources"]` into `per_source`, copy `stats["failed_sources"]` into `failed_sources`, compute `duration_ms` from existing field). `await crawl_run_repo.insert(record)`. If the insert raises, log `_log.exception("crawl_run.persist_failed", ...)` and re-raise — failing to record a crawl run is a real defect, not soft-failed. Append the new `crawl_run_id` (return value of insert) into `stats` as `stats["crawl_run_id"] = str(crawl_run_id)` so the CLI can echo it.
- **acceptance_criteria**:
  - `crawler run-once --top-n 1` echoes a line containing `crawl_run_id=<uuid>`.
  - One row per CLI invocation appears in `crawl_runs`. Two back-to-back invocations produce 2 distinct rows; `started_at` of row 2 > `finished_at` of row 1.
  - `failed_sources` is `{}` (empty array) on success.
- **commit**: `feat(03-01-T06): orchestrator writes crawl_runs row at end of run_once`
