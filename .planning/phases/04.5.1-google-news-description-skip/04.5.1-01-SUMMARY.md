## Why this phase existed

v0.5.0 (Plan 04.5-01) recovered descriptions for ~47/77 production topics by
backfilling from `raw_payload->>'summary'`. Post-deploy smoke (operator
inspection of live `/api/topics` payload + Cloud Run logs) revealed that
Google News RSS `<description>` is **structurally an `<ol><li><a>`
related-articles HTML fragment** — links to similar stories the aggregator
clustered — not publisher prose. ~29 of the first 30 inspected production
topics had this HTML-link-list content masquerading as a description.

Root cause: not a parser bug. The RSS parser was honestly extracting what
Google chose to put in `<description>`. The bug is upstream of the parser:
the data shape itself is wrong for our use case.

NYT homepage RSS `<description>` is real prose standfirst. HN has no
description by design. Only Google News needs suppression.

Operator decision (m0141): **Option A — strip at parser level**, reject
HTML-detection heuristics (B) and accept-as-is (C). Clean, deterministic.

## What shipped

### T01 — Parser opt-out flag (`63ec8f6`)

`RssSource.__init__` gained `capture_summary: bool = True`. When False,
parser forces `RawItem.description = None` regardless of feed content. Raw
value still preserved in `raw_payload['summary']` for forensic fidelity —
only the topic-level description column is suppressed.

`_RSS_SOURCES` tuple shape extended from `(name, url)` to
`(name, url, capture_summary)`. Registration:

- `nyt_homepage`: `True` (default)
- `google_news`: `False`

2 new unit tests on `test_rss_adapter.py`:

- `capture_summary=False` → all `description is None`, `raw_payload['summary']` still present
- default `True` → descriptions preserved (regression guard)

**Suite:** 48/48 PASS (test_cross_source_dedup pre-existing skip carried).

### T02 — Idempotent cleanup script (`c85e9b8`)

`scripts/clear_google_news_descriptions.py`. Same async-engine + `--dry-run`
pattern as `scripts/backfill_descriptions.py` (Plan 04.5-01 T05).

Predicate: NULL `topics.description` if and only if **every** observed source
for that topic is `google_news`. Cross-source topics (observed by both NYT
and Google News, where first-non-empty merge already preserved NYT prose)
keep their description — only Google-News-only rows are touched.

Verified against local DB:

- dry-run: 27 rows would be NULLed
- real run: 27 rows NULLed
- re-run: 0 rows (idempotent ✅, `WHERE description IS NOT NULL` guard fires)
