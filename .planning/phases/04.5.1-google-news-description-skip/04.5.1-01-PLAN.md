# Phase 4.5.1 — Google News Description Skip

**Status:** executing inline (single-commit fix, no separate discuss-phase needed)
**Trigger:** post-v0.5.0 production smoke (m0140) revealed Google News `<description>` is upstream-shaped as an `<ol><li><a>` related-articles HTML fragment, not publisher prose. NYT standfirsts are correct prose. HN has no description by design.
**Operator decision (m0141):** Option A — strip Google News descriptions entirely at the parser level. Cleanest, deterministic, no heuristics. NYT descriptions preserved.

## Scope

**IN:**
1. Add `capture_summary: bool = True` to `RssSource.__init__`. When False, the parser writes `description=None` on `RawItem` regardless of feed content. The raw value still lands in `raw_payload['summary']` (untouched for forensics — does not pollute the topic-level description column).
2. Update `_RSS_SOURCES` in `composition.py` to a `(name, url, capture_summary)` triple. NYT homepage keeps True. Google News set to False.
3. Add unit test: Google News-shaped feed → `RawItem.description is None`; NYT-shaped feed → `RawItem.description == prose`.
4. One-shot idempotent cleanup script `scripts/clear_google_news_descriptions.py` that NULLs `topics.description` for topics whose ONLY observed source is `google_news` (so cross-source topics already merged with NYT prose retain that prose under the first-non-empty rule). Re-runnable safely.
5. SUMMARY + ROADMAP closeout + cloudbuild image tag bump to v0.5.1.

**OUT:**
- HTML-stripping heuristics (rejected option B).
- Storing the Google News related-articles fragment anywhere user-visible.
- Schema changes (none needed).
- New Alembic migration (none needed).

## Constraints preserved
- ARC-001 (no AI in ingest).
- No new outbound HTTP fetches.
- Single Alembic tree in `packages/core` (no changes to 0001-0004).
- Backwards-compatible: `capture_summary` defaults to True so other RSS sources added later get descriptions automatically (opt-out, not opt-in).
- Idempotent cleanup script (`WHERE description IS NOT NULL` join filter).

## Tasks (3, autonomous, inline)
- **T01**: Parser + composition change (`rss.py` + `composition.py`) + unit test. Single commit.
- **T02**: One-shot cleanup script (`scripts/clear_google_news_descriptions.py`).
- **T03**: SUMMARY + ROADMAP + STATE + cloudbuild v0.5.1 bump. Operator runs `gcloud builds submit` post-execute.

## Decisions locked
- **D1 (skip mechanism):** boolean flag on the adapter, opt-out at registration site. Considered conditioning inside `rss.py` on `self.name == 'google_news'` — rejected because it couples the generic adapter to a specific source name.
- **D2 (cleanup scope):** only NULL `topics.description` when ALL sources for that topic are `google_news`. If the topic was ever co-observed with NYT and merged with NYT's prose, keep that prose (first-non-empty merge semantics).
- **D3 (no separate discuss-phase):** scope is unambiguous (operator already picked A); proceed straight to plan-execute consolidated in this doc to minimise audit-trail overhead for a ~10-line fix.
