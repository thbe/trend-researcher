# Plan 02-03: Orchestrator Hardening + Cross-Source Dedup (Wave 3)

**Phase**: 2 — Multi-Source Ingest
**Mode**: mvp
**Depends on**: Plan 02-02

## Goal
Two pieces of behavior that **only matter once we have multiple sources** become testable now:
1. **Per-source failure isolation**: one source raising mid-fetch must not abort the whole run.
2. **Cross-source dedup**: a story appearing on NYT and Google News must collapse to one `topic` row with two `topic_sources` rows.

No new adapters in this plan — only orchestrator behavior + a dedup proof.

## Test discipline
Both tasks follow strict TDD: write the failing test first, run it RED, then implement / verify GREEN.

## Tasks

### T01 — Orchestrator: per-source failure isolation (TDD)
- **Step 1 — failing test**: append to `services/crawler/tests/test_orchestrator.py` (create file if it doesn't exist; if it does, append):
  - `FakeSource(name, items_or_exc)`: when `items_or_exc` is a list, returns it; when an exception, raises it.
  - `FakeRepository`: in-memory list of inserts and updates, no DB.
  - Test `test_one_source_failure_does_not_abort_run`:
    - Sources: `[FakeSource("good_a", [item_a])`, `FakeSource("bad", RuntimeError("boom"))`, `FakeSource("good_b", [item_b])]`.
    - Run `run_once_async(sources, repo, top_n=10)`.
    - Assert: result completes (no exception bubbles out).
    - Assert: repo received `item_a` and `item_b` inserts.
    - Assert: result stats include something indicating the failure (e.g. `failed_sources == ["bad"]` or `errors == 1`).
  - Test `test_all_sources_succeed_no_failed_sources_field`:
    - Two healthy fake sources, assert `failed_sources` is empty / `errors == 0`.
  - Run `uv run --package crawler pytest tests/test_orchestrator.py -v` → tests FAIL or skip (RED).
- **Step 2 — implement**: edit `services/crawler/src/crawler/app/orchestrator.py`:
  - Wrap each `await source.fetch(...)` in `try/except Exception as exc` (broad — we genuinely want to swallow + continue).
  - On failure: structlog `_log.exception("crawler.source_failed", source=source.name)` and append `source.name` to a `failed_sources: list[str]`.
  - Extend the result dict / dataclass returned by `run_once_async` with `failed_sources: list[str]` (and update the `crawl.complete` JSON log to include it).
  - Backward compatibility: Phase 1 tests that don't reference `failed_sources` must still pass.
- **Step 3 — verify GREEN**: `uv run --package crawler pytest tests/test_orchestrator.py -v` → both new tests pass; `uv run --package crawler pytest -v` → full suite green.
- Commit: `feat(02-03-T01): orchestrator per-source failure isolation (TDD)`

### T02 — Cross-source dedup proof (TDD against the real repo)
- **Step 1 — failing test**: file `services/crawler/tests/test_cross_source_dedup.py`
  - Skip-gated on `TEST_DATABASE_URL` (matches existing repo-test pattern in `conftest.py`).
  - Test `test_same_headline_from_two_sources_collapses_to_one_topic`:
    - Build two `RawItem`s with the SAME title (e.g. `"Major retailer announces price cuts"`) but different `source_name` (`nyt_homepage` vs `google_news`) and different `url`.
    - Drive them through the real `run_once_async` against a real `SqlAlchemyTopicRepository` + an in-memory or test-DB session, with two `FakeSource` instances each returning one of the items.
    - Assert: `SELECT count(*) FROM topics` → 1.
    - Assert: `SELECT count(*) FROM topic_sources WHERE topic_id = <that topic>` → 2.
    - Assert: the `source_name` set on those two rows is `{"nyt_homepage", "google_news"}`.
    - Assert: `topics.observation_count` == 2 (the second observation bumped the counter via `_bump_topic`).
  - Test `test_same_headline_same_source_does_not_create_duplicate_source_row`:
    - Both `RawItem`s have same title, same `source_name`, same `url`, same `observed_at` → should hit the `IntegrityError` branch in `update_existing` and the source-row count must be exactly 1, but `observation_count` must still be 2.
  - Run `uv run --package crawler pytest tests/test_cross_source_dedup.py -v` (with `TEST_DATABASE_URL` exported) → both tests RED initially? They might already pass because the Phase 1 dedup is source-agnostic. **That's the proof.** If they pass on first run, that confirms the existing code already supports cross-source dedup with no changes required.
- **Step 2 — only if tests fail**: investigate which assumption broke and fix in the dedup or repository layer (no schema changes — that's a Phase 2 invariant). Most likely cause if it fails: the `find_candidates` recent-window scan returned 0 rows because the test seeded nothing — fix by ensuring both items go through `run_once_async` sequentially in the same session lifecycle.
- **Step 3 — verify GREEN**: tests pass with `TEST_DATABASE_URL`; tests skip cleanly without it.
- Commit: `test(02-03-T02): cross-source dedup proof (TDD; expected to pass with no impl change)`
