# Plan 04-01: `v_topic_stats` Postgres VIEW (Wave 1)

**Phase**: 4 — Topic API & UI Shell
**Mode**: mvp
**Depends on**: nothing (greenfield migration in Phase 4)
**REQ-IDs in scope**: STO-006 (breadth + longevity derived, not stored)

## Goal
Ship a read-only Postgres VIEW `v_topic_stats` that exposes `(topic_id, breadth, longevity_seconds)` derived from `topics` + `topic_sources`. After this plan, any SQL/ORM caller can `SELECT breadth, longevity_seconds FROM v_topic_stats WHERE topic_id = ?` instead of re-deriving COUNT(DISTINCT)/EXTRACT inline. STO-006 is satisfied at the storage layer — nothing is denormalised into `topics`.

## Slice shape (vertical)
Alembic migration → Postgres VIEW → integration test that proves the view's SQL semantics. No ORM model, no route — those land in 04-02 / 04-03.

## Threat model (per §5.55)
- Read-only VIEW; no INSERT/UPDATE/DELETE path. CREATE VIEW runs at migration time as the migration role (already has DDL rights from Phase 1).
- View leaks the same data already in `topics` + `topic_sources` (no PII beyond what's already there). No new attack surface.
- `EXTRACT(EPOCH FROM ...)` on `(last_seen_at - first_seen_at)`: cannot overflow `bigint` for realistic data (would require ~292 billion years).
- Topic with zero sources: LEFT JOIN keeps the row, `COUNT(DISTINCT ts.source_name) = 0`. Acceptable (topic exists but unobserved — edge case, but a real possibility if a future code path inserts a Topic before its first TopicSource).

- `upgrade()` executes `op.execute(...)` with this exact SQL:
  ```sql
  CREATE VIEW v_topic_stats AS
  SELECT
    t.id AS topic_id,
    COUNT(DISTINCT ts.source_name) AS breadth,
    COALESCE(EXTRACT(EPOCH FROM (t.last_seen_at - t.first_seen_at))::bigint, 0) AS longevity_seconds
  FROM topics t
  LEFT JOIN topic_sources ts ON ts.topic_id = t.id
  GROUP BY t.id;
  ```
  COALESCE guards the (impossible-today, but defensive) case where `first_seen_at` or `last_seen_at` is NULL — they are NOT NULL in schema, but COALESCE is cheap insurance against a future schema relaxation.
- `downgrade()` executes `op.execute("DROP VIEW IF EXISTS v_topic_stats;")` (IF EXISTS for safe re-runs).

## Must-haves (goal-backward verification)
- After this plan, `SELECT breadth, longevity_seconds FROM v_topic_stats` is the **single source of truth** for these two derived numbers. No route, no ORM, no test re-derives them inline (enforced by code-review when 04-02/04-03 land).
- STO-006 is honoured at storage (`topics` schema has zero new columns).
- The view formula is one Alembic migration, version-controlled, with a rollback path. Inline subqueries in routes are explicitly rejected by this plan (would invalidate the "reused by Phase 5/8" payoff).
- `longevity_seconds` is in seconds (DB unit, per CONTEXT.md G4 sub-decision). Human-readable formatting is the frontend's job (04-04 `formatLongevity`).
