# Plan 04-02: `/api/*` re-prefix + `GET /api/topics` list (Wave 2)

**Phase**: 4 — Topic API & UI Shell
**Mode**: mvp
**Depends on**: Plan 04-01 (`v_topic_stats` view must exist for `breadth` + `longevity_seconds` to be queryable)
**REQ-IDs in scope**: UI-002 (sortable topic browsing), STO-006 (derived breadth+longevity served from view)

## Goal
Two things, intentionally bundled (re-prefix is ~5 LoC and the topics list immediately needs to live at `/api/topics` — splitting them creates a no-op intermediate state):
1. **Re-prefix** Phase 3's `/healthz` and `/runs` under `/api/*` so the future `StaticFiles` catch-all at `/` (05-05) doesn't swallow them.
2. **Add `GET /api/topics`** — wrapper-shape response with `topics: [...]`, `limit`, `sort`. Sort whitelist `{breadth, longevity, last_seen_at}` with leading `-` for desc, default `-last_seen_at`, non-whitelisted → 400.

After this plan, `curl localhost:8000/api/topics?sort=-breadth&limit=5` returns the 5 topics with the most distinct source names, JSON-shaped.

## Slice shape (vertical)
HTTP route → handler → SQLAlchemy JOIN against `topics` + `v_topic_stats` → Pydantic response → JSON. End-to-end through the existing FastAPI shell.

## Threat model (per §5.55)
- Read-only endpoint on the same internal-only port 8000 as `/healthz` + `/runs`. No auth in v1 (ARC-004).
- `sort` is a string parameter — whitelisted server-side via an explicit `{breadth, longevity, last_seen_at}` set; non-matches return 400. No SQL identifier interpolation: the whitelist maps `str → InstrumentedAttribute` server-side (no f-string, no `text()` with the user value).
- `limit` clamped to `[1, 100]` via FastAPI `Query(20, ge=1, le=100)` — identical to `/runs`.
- Response data: `topics.title`, `topics.description` — both already-public source content (news headlines). No PII.
- Re-prefix: existing operator scripts/bookmarks pointing at `/healthz` and `/runs` break. README + smoke script updated in this plan; one-line breadcrumb in `/healthz` route docstring explains the prefix move so a future grep finds it.

## Must-haves (goal-backward verification)
- All four routes (`/api/healthz`, `/api/runs`, `/api/topics`, future `/api/topics/{id}`) live under `/api/*` — uniform prefix, no root-mounted exceptions.
- `GET /api/topics` reads `breadth` + `longevity_seconds` from `v_topic_stats` (Plan 04-01) — **never** re-derived inline.
- Sort whitelist matches CONTEXT.md G5 exactly: `{breadth, longevity, last_seen_at}`. Adding a future field requires editing one constant + one test.
- Response shape is `{topics: [...], limit, sort}` — wrapper object, not bare array. Mirrors `RunsListResponse` (Phase 3 locked idiom).
- Sort tiebreaker is deterministic (`topics.id`) — pagination over identical sort values is stable.
- Phase 3's `/healthz` + `/runs` behaviour identical at `/api/healthz` + `/api/runs`; only the path changed.

## Wave / dependency context
Wave 2, depends on 04-01 (view). 04-03 (Wave 3 — detail endpoint) inherits this plan's `v_topic_stats` access pattern.

## Out of scope (explicit)
- Filter params `?since=`, `?source=`, `?breadth_gte=` (Phase 5+).
- Offset/cursor pagination (only `?limit` per CONTEXT.md).
- Topic-detail endpoint (Plan 04-03).
- `topic_metadata` or `sources` on list-row projection (kept for detail).
- ORM model for `v_topic_stats` (raw `Table(..., autoload=False)` declaration is sufficient and avoids a write-surface trap).
