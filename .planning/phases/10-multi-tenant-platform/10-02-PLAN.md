# Plan 10-02: Per-Department Scoping + Source Subscription (Wave 2)

**Phase**: 10 — Multi-Tenant Market Intelligence Platform
**Mode**: mvp
**Depends on**: 10-01 (departments + RBAC + active-department dependency)
**REQ-IDs in scope**: MT-003, MT-004, MT-006, MT-008, MT-009
**Autonomous**: false — biggest schema change in this phase. FKs added to
hot tables; backfill must succeed atomically; operator gate before prod
migration.

## Goal
Take every currently-global stateful concept and pin it to a department:
- `ai_config` becomes per-department (PK shift from `key` to `department_id`)
- `business_cases` gains `department_id` FK + composite unique constraint
  including dept
- `assessment_jobs` gains `department_id` FK
- New `department_sources` table holds per-dept source subscription;
  `crawl_config.enabled` column is dropped (single source of truth)
- Crawler orchestrator now computes the effective source list as the
  **union of all departments' enabled subscriptions**
- All affected API routes resolve the active department via the dependency
  from 10-01 and filter/write accordingly
- `topics` and `topic_sources` REMAIN GLOBAL (MT-003)

After this plan: a logged-in user with active dept = Default sees identical
behaviour to pre-Phase-10. A second department can be created (10-01) and
its members will see *their own* config and assessments, isolated from
Default's.

`structured_output` column on business_cases is deferred to 10-03 (lands
with the framework refactor — adding it here would require a no-op
framework_id which adds confusion).

## Slice shape (vertical)
Single Alembic migration with multiple table reshapes + backfill in one
transaction → ORM mirror → core tests for the migration → crawler change +
its unit test → API re-scope (5 routes) + 1 new route → API tests.

## Threat model
- **Backfill** runs in the upgrade transaction. If any step fails, all roll
  back. Single-tenant prod has exactly one row in `ai_config` and a known
  set of business_cases / assessment_jobs / crawl_config rows — backfill
  is straightforward.
- **Reshaping ai_config**: changing PK from `key` to `department_id` is
  destructive. Use a "create new, copy, drop old, rename" sequence within
  one migration.
- **Composite UNIQUE on business_cases**: existing rows may already violate
  the proposed constraint if the same topic was assessed twice with the
  same prompt_version + model. In single-tenant prod the user has
  re-assessed topics; we MUST verify no in-flight duplicates before adding
  the constraint, otherwise migration fails. Plan T01 includes a pre-check.
- **Crawler orchestrator change**: if no department has subscribed to any
  source (e.g., migration glitch), crawler runs zero sources silently. Add
  a log warning + safe-default: if `department_sources` is empty, log
  warning and fall back to all sources from `crawl_config` (defensive).
- **Header dependency** is now required by formerly-global endpoints —
  any caller (SPA, scripts/smoke_phase*.sh) that omits the header will
  4xx. Update the smoke scripts in this plan.

**upgrade() steps (single transaction, ordered):**

1. **Pre-check business_cases for duplicate violations:**
   ```python
   conn.execute(sa.text("""
     SELECT topic_id, prompt_version, model_used, COUNT(*) AS c
     FROM business_cases
     GROUP BY topic_id, prompt_version, model_used
     HAVING COUNT(*) > 1
   """))
   ```
   If any row returned: raise RuntimeError with message instructing
   operator to dedup before migrating. (Real production has none expected
   — but we want a hard failure, not silent constraint violation.)

2. **Create `department_sources` table:**
   - `department_id` UUID FK → departments.id ON DELETE CASCADE
   - `source_name` TEXT
   - `enabled` BOOLEAN NOT NULL default false
   - `created_at`, `updated_at` TIMESTAMPTZ
   - PK `(department_id, source_name)`
   - Index on `source_name` (for "which depts use hackernews?" queries
     and for crawler's union query)

3. **Backfill `department_sources` from `crawl_config`:**
   - For every (existing `crawl_config.source_name`, `Default
     department.id`), insert `department_sources` row with
     `enabled = crawl_config.enabled` for that source.

4. **Drop `crawl_config.enabled` column.** Update the model accordingly in
   T02. Keep all other columns (top_n, capture_summary, verify_ssl,
   feed_url) — they're tech config, not dept preference.

5. **Reshape `ai_config`:**
   - `op.create_table("ai_config_new", ...)` with all current columns BUT
     PK shifted: `department_id UUID PRIMARY KEY FK → departments.id ON
     DELETE CASCADE`. No `key` column.
   - `INSERT INTO ai_config_new (department_id, base_url, model, api_token,
     business_context, opportunity_criteria, risk_criteria,
     thinking_effort, request_timeout_seconds, updated_at) SELECT
     '<default_uuid>', base_url, model, ... FROM ai_config WHERE
     key='default'`. (Hardcode the default-dept UUID from 10-01.)
   - `op.drop_table("ai_config")`
   - `op.rename_table("ai_config_new", "ai_config")`

6. **Extend `business_cases`:**
   - `op.add_column("business_cases", sa.Column("department_id", UUID,
     sa.ForeignKey("departments.id", ondelete="CASCADE"), nullable=True))`
     — temporarily nullable for backfill.
   - `UPDATE business_cases SET department_id = '<default_uuid>'`
   - `op.alter_column("business_cases", "department_id", nullable=False)`
   - Index `business_cases (department_id)`.
   - **Note:** the UNIQUE (topic_id, department_id, framework_id,
     prompt_version, model_used) constraint is added in **10-03** because
     `framework_id` doesn't exist yet. Add a partial constraint now? No —
     defer. Document in the docstring.

7. **Extend `assessment_jobs`:**
   - Same pattern: add nullable FK, backfill to default dept, set NOT NULL.
   - Index on `department_id`.

8. **downgrade()** reverses each step. For step 5 (ai_config reshape), the
   reverse is the same dance in reverse. Don't try to preserve
   per-department configs on downgrade — single-tenant downgrade picks
   Default's config and that's it. Document this.
</action>
