# Plan 10-04: Frontend — Multi-Tenant SPA Shell (Wave 4)

**Phase**: 10 — Multi-Tenant Market Intelligence Platform
**Mode**: mvp
**Depends on**: 10-03 (assessment_frameworks API + per-dept default endpoints
exist; business_cases responses carry `structured_output` + framework metadata)
**REQ-IDs in scope**: MT-002 (dept switcher + role-gated nav), MT-005
(framework picker UI), MT-007 (structured output rendering), MT-010 (rename
hook)
**Autonomous**: true — pure frontend, no schema or backend changes. Behind
the existing auth, low blast radius.

## Goal
Replace the temporary `localStorage`-only header injection (10-02 T08) with a
proper multi-tenant SPA shell:
- Pinia stores for session + active department + frameworks.
- Department switcher in the app bar; persisted across reloads; calls
  `POST /api/auth/switch-department` to refresh cookie.
- Role-gated nav (NavDrawer / AppBar action items hidden by current role +
  superadmin).
- Departments admin view (superadmin) for CRUD + member management.
- Per-department settings views: AI config (per dept), source subscriptions
  (per dept), framework picker (per dept).
- Three framework display cards (`VerdictCard`, `SwotCard`, `PestleCard`)
  driven by `business_case.framework.display_component`. Wrapper
  `BusinessCaseCard` dispatches.
- Framework picker on the Assess action so the operator chooses which
  framework runs (defaulted from dept default).
- OpenAPI codegen (`openapi-typescript`) wired into the build, output to
  `web/src/api/generated/`. The runtime client (`client.ts`) stays
  hand-written; types feed call sites for compile-time safety.
- `web/src/lib/strings.ts` collects every UI string that mentions the
  product name / market — single file for the eventual rename (G8).

After this plan: a logged-in user with multiple departments can switch
between them, see role-appropriate UI, configure each dept independently,
and view assessment results in framework-appropriate cards.

## Slice shape (vertical)
Tooling (Pinia + openapi-typescript install + codegen script) → stores →
shell components (AppBar, NavDrawer, DepartmentSwitcher) → framework cards
& wrapper → settings views → re-scoped existing views → router guards →
strings extraction → tests.

## Threat model
- **Active dept drift between cookie and store**: the cookie carries the
  authoritative `active_department_id` (set by 10-01's
  `switch-department` endpoint). The Pinia store is mirror state. If they
  drift (e.g., user opens two tabs and switches in one), API calls will
  4xx with a clear "active dept mismatch" error. Mitigation: API client
  interceptor handles 401/409 on dept mismatch by refetching `/api/auth/me`
  to rehydrate store; if still mismatched, force the switcher modal open.
- **Role-gated nav hiding ≠ authorization**: hiding the "Assess" button for
  `viewer` does NOT prevent API calls. Backend (10-02) enforces. UI gating
  is UX only. Documented in `roles.ts`.
- **OpenAPI drift**: codegen runs at build time, not at lint/CI gate (yet).
  If backend changes a response shape and frontend devs forget to regenerate,
  types lie. Mitigation: add `npm run gen:api` to the dev `predev` hook;
  document in README. CI gate is a Phase 11 candidate.
- **Strings file as bottleneck**: collecting every product-name string into
  `strings.ts` is mechanical but easy to miss new strings later. Mitigation:
  add a `lint` ESLint rule (custom or via `eslint-plugin-i18next`-style
  pattern) flagging hardcoded "Trend"/"Retail" literals in `.vue` /`.ts`
  files outside `strings.ts`. Defer rule itself if heavy; document
  convention in README.
- **Vuetify 3 / Pinia interaction**: Pinia setup happens in `main.ts` before
  the Vuetify plugin; verify no ordering bug (Vuetify components inside
  Pinia setup hooks must be safe).

Create three Pinia stores:

**`web/src/stores/session.ts`:**
- State: `user: {id, username, is_superadmin} | null`,
  `departments: Array<{id, name, role}>`, `activeDepartmentId: string | null`,
  `activeRole: 'viewer'|'analyst'|'dept_lead'|null`.
- Actions:
  - `async hydrate()` — calls `GET /api/auth/me`; populates state. Called on
    app mount.
  - `async switchDepartment(id: string)` — calls `POST /api/auth/switch-department`,
    updates `activeDepartmentId` + `activeRole`, persists chosen id to
    `localStorage['activeDepartment']` for cross-tab continuity. Triggers a
    refetch of `frameworks` store (T04 dependency).
  - `logout()` — calls `POST /api/auth/logout`, clears all stores + localStorage,
    redirects to `/login`.
- Getters:
  - `isSuperadmin`
  - `canAssess` — role in `('analyst', 'dept_lead')` || superadmin
  - `canEditDeptConfig` — role === 'dept_lead' || superadmin
  - `canHarmonize` — role === 'dept_lead' || superadmin
  - `canManageMembers` — role === 'dept_lead' || superadmin
  - `canManageDepartments` — superadmin

**`web/src/stores/activeDepartment.ts`:**
- Thin re-export of `session.activeDepartmentId / activeRole / canX` getters
  for ergonomic use (or merge into session if it feels cleaner — pick by
  developer taste at implementation time). If kept separate, document why.
- *Decision at implementation:* if all consumers naturally call
  `useSessionStore()`, drop `activeDepartment.ts` and remove from
  `files_modified`. Don't create dead modules.

**`web/src/stores/frameworks.ts`:**
- State: `system: Framework[]` (cached from `GET /api/frameworks`),
  `mine: {enabled: Framework[], defaultId: string} | null` (per active dept).
- Actions:
  - `async loadSystem()` — once per session.
  - `async loadMine()` — on dept change; called by `session.switchDepartment`.
  - `async updateMine({enabled, defaultId})` — calls `PUT /api/frameworks/mine`.
- Getters: `defaultFramework`, `byId(id)`, `displayComponentFor(id)`.

Update `web/src/api/client.ts`:
- Remove the 10-02 T08 `localStorage`-only injection. Replace with: read
  `useSessionStore().activeDepartmentId` and inject `X-Active-Department`
  header on every request. Pinia exposed via `getActivePinia()` so the client
  works outside Vue setup context.
- Response interceptor: on 401 → redirect to `/login`. On 409 with body
  `{detail: "active department mismatch"}` → call `session.hydrate()` once,
  then retry the original request once. On second 409, surface to user.
- Use generated types from `src/api/generated/api.ts` for the wrapper method
  signatures where ergonomic; full migration to generated types is out of
  scope (call sites stay loose for v1).
</action>

- Refreshing the page re-hydrates from `/api/auth/me` (cookie persisted).
- `client.ts` injects `X-Active-Department` on every request automatically.
