# Plan 10-03: Pluggable Assessment Frameworks (Wave 3)

**Phase**: 10 — Multi-Tenant Market Intelligence Platform
**Mode**: mvp
**Depends on**: 10-02 (business_cases has `department_id`; ai_config per-dept)
**REQ-IDs in scope**: MT-005 (frameworks pluggable per-dept), MT-007 (structured output)
**Autonomous**: false — touches the assessor's prompt/parse pipeline. Operator
gate before production rollout because verdict-framework behaviour must be
preserved bit-for-bit to keep existing dashboards meaningful.

## Goal
Make the assessor framework-aware:
- New `assessment_frameworks` table holds the three seeded frameworks
  (`verdict`, `swot`, `pestle`) with `key`, `name`, `display_component`,
  `json_schema` (JSONB), `prompt_version` (str).
- New `department_frameworks` join table: which frameworks a dept has enabled,
  one marked as `is_default` per dept.
- `business_cases` gains `framework_id` FK + `structured_output` JSONB +
  composite UNIQUE `(topic_id, department_id, framework_id, prompt_version,
  model_used)`.
- Assessor refactored: one Python module per framework owning `build_prompt`,
  `parse_output`, `JSON_SCHEMA`. Pipeline looks up framework by id, dispatches.
- `verdict` framework MUST produce **identical** prompts and parse outputs to
  the current pre-10-03 assessor — single-tenant prod assessments stay
  reproducible.
- API: `GET /api/frameworks` (system list), `GET/PUT /api/frameworks/mine`
  (dept's enabled set + default), `POST /api/assessment/...` accepts optional
  `framework_id` (falls back to dept default).

After this plan: Default dept still runs verdict by default, identical to
today. Operators can now enable SWOT or PESTLE on any dept and run those.

## Slice shape (vertical)
Migration (table + columns + backfill + UNIQUE) → ORM mirror → seed function →
core tests → assessor framework modules + registry → pipeline refactor →
assessor tests (golden-file verdict identity test included) → API routes +
schemas → API tests.

## Threat model
- **Verdict-identity regression**: the single biggest risk. Existing dashboards
  read `relevance_verdict` / `relevance_reason` / etc.; if the refactored
  verdict framework changes wording or parsing by even one character, the next
  re-assessment will look like a model drift to operators. Mitigation: T05
  includes a **golden-file test** — captured current prompt output for a fixed
  topic fixture must match byte-for-byte. T07 validates parse_output on a
  captured `raw_response` from prod and asserts the structured_output JSON
  round-trips back to the same `relevance_verdict` / `relevance_reason` /
  `importance_score` / etc.
- **JSON schema validation failure**: an LLM that returns malformed JSON for
  SWOT/PESTLE breaks the pipeline. The pipeline currently has best-effort
  parsing for verdict (regex + fallback). New frameworks need: validate
  against `JSON_SCHEMA` → on failure, retry once with `system` correction
  prompt → on second failure, mark `assessment_jobs` row as `failed` with
  `error_message` and write nothing to business_cases. Keep job atomicity.
- **Composite UNIQUE backfill**: 10-02 already verified no duplicate
  `(topic_id, prompt_version, model_used)` exist. After adding
  `framework_id = verdict.id` to every row, the new constraint
  `(topic_id, department_id, framework_id, prompt_version, model_used)` is
  strictly weaker → automatically satisfied. T01 includes a post-backfill
  assertion query before adding the constraint.
- **`structured_output` backfill on existing rows**: must be computed from
  current columns. Build a dict matching verdict's JSON schema. Any row with
  NULL `relevance_verdict` (failed assessment) gets `structured_output =
  '{}'::jsonb` and is flagged with a comment in the migration.
- **Department default framework race**: enabling/disabling a framework while
  an assessment job is in flight referencing it. Mitigation: only require that
  `framework_id` exists in `assessment_frameworks` at job creation time; the
  drain loop doesn't re-check dept enablement. Document this trade-off
  (operator may disable a framework but in-flight jobs still complete using it).
