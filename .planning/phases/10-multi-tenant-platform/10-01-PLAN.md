# Plan 10-01: Departments + RBAC Foundation (Wave 1)

**Phase**: 10 — Multi-Tenant Market Intelligence Platform
**Mode**: mvp
**Depends on**: 10-00 (REQUIREMENTS.md MT-* entries minted)
**REQ-IDs in scope**: MT-001, MT-002, MT-008
**Autonomous**: false — operator gate before applying the migration in
production. This is the first schema change and brings the active-department
header concept online; worth a human review pass.

## Goal
Land the foundational multi-tenant primitives:
1. `departments` table + ORM
2. `users.is_superadmin` boolean column
3. `user_departments` join table with `role` enum
4. Default-department seed migration that promotes the existing seed user to
   `is_superadmin = true` and `dept_lead` of Default
5. `get_active_department()` FastAPI dependency (reads
   `X-Active-Department` header, verifies membership, returns
   `(department, role)`)
6. `/api/departments/*` CRUD + member-management endpoints (superadmin for
   CRUD, dept_lead for member mgmt within their dept)
7. Login response extended with the user's department list + role per dept

After this plan: no existing endpoint is broken (active-dept dependency
exists but isn't yet *required* by ai_config / crawl_config / assessment /
dashboard — that's 10-02's job). Superadmin can create depts, add users with
roles, and the SPA can populate a switcher (Even though there's nothing for
it to switch yet — 10-04 builds the UI).

## Slice shape (vertical)
Migration → ORM → tests (per-package) → API dependency + routes → API tests.
The "vertical-ness" is end-to-end *for the new endpoints only*; existing
endpoints are deliberately untouched (decoupled in 10-02 to bound blast
radius of this plan).

## Threat model
- New `departments` + `user_departments` tables are write surfaces;
  validation is server-side (Pydantic schemas).
- `is_superadmin` is a privilege escalation vector — only modifiable via
  Alembic or a superadmin-only endpoint (no self-promotion path).
- `X-Active-Department` is operator-supplied via the SPA; the dependency
  MUST verify membership server-side. Trusting the header value would let
  any authenticated user access any dept's data. Test for this explicitly.
- Default-dept migration runs as part of `alembic upgrade head` — if it
  fails partway, the migration must roll back atomically (single
  transaction). Alembic guarantees this on PG when no DDL outside the
  transaction.

- `downgrade()`:
  1. `op.drop_table("user_departments")`
  2. `op.drop_column("users", "is_superadmin")`
  3. `op.drop_table("departments")` (the seed rows go with it)

  current_user: User = Depends(...), session = Depends(get_session)) ->

  - Read header `X-Active-Department` (department UUID).
  - If `is_superadmin` and header absent → 400 "active department required".

    membership (deterministic order: oldest first); if zero memberships →
    403 "user has no department memberships".
  - Query `user_departments` for `(current_user.id, header_value)`. If no

    implicit role of `dept_lead` even without membership (so they can
    administer any dept).
  - Return `ActiveDepartment(department, role)`.
- Add helper `require_role(*allowed: RoleLiteral)` that returns a
  dependency factory: `def dep(ad: ActiveDepartment = Depends(
  get_active_department)) -> ActiveDepartment: if ad.role not in allowed
  and not ad.is_superadmin_override: raise HTTPException(403)`.
