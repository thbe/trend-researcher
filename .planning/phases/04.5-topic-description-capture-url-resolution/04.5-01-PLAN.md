    Create a Python one-shot backfill script per locked D-Q2. Idempotent (safe to run twice). Operator-invoked via `docker compose exec api python -m scripts.backfill_descriptions` (dev) or `gcloud run jobs execute` later. Not in the crawl path.

    Script logic (two independent passes, both inside one transaction per row):

    **Pass 1 — backfill topics.description (167 production rows):**
    ```sql
    UPDATE topics t
    SET description = sub.summary, updated_at = now()
    FROM (
      SELECT DISTINCT ON (ts.topic_id) ts.topic_id, ts.raw_payload->>'summary' AS summary
      FROM topic_sources ts
      WHERE ts.raw_payload->>'summary' IS NOT NULL
        AND length(trim(ts.raw_payload->>'summary')) > 0
      ORDER BY ts.topic_id, ts.observed_at ASC  -- earliest observation per topic = first-non-empty
    ) sub
    WHERE t.id = sub.topic_id
      AND t.description IS NULL;  -- idempotent guard
    ```

    **Pass 2 — backfill topic_sources.resolved_url for google_news rows:**
    Select all rows where `source_name = 'google_news' AND resolved_url IS NULL AND url LIKE 'https://news.google.com/rss/articles/CBM%'`. For each, call `decode_google_news_url(row.url)` (import from `crawler.adapters.sources.google_news_url`). If decode returns a string, `UPDATE topic_sources SET resolved_url = :decoded WHERE id = :row_id`. If None, skip (the structlog warn already fired inside the decoder).

    Print a final summary to stdout: `backfilled descriptions: N, backfilled resolved_urls: M, skipped (decoder failed): K`. Exit code 0 on success, 1 on any uncaught DB exception.

| Threat ID  | Category            | Component                           | Disposition | Mitigation Plan                                                                                                                                |
| ---------- | ------------------- | ----------------------------------- | ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------- |

| T-04.5-02  | Denial of Service   | T05 backfill script                 | mitigate    | `WHERE description IS NULL` and `WHERE resolved_url IS NULL` guards make script a no-op on second invocation; cannot deadlock or duplicate-write. |
| T-04.5-03  | Information Disclosure | structlog "google_news.decode_failed" | mitigate    | Log only first 80 chars of URL (CBM tokens are public anyway, but bound the log line for budget hygiene).                                       |
| T-04.5-04  | Tampering (XSS)     | TopicList/TopicDetail description rendering | mitigate    | Use Vue's default text interpolation (`{{ topic.description }}`) — never `v-html`. Vue escapes by default; description never reaches DOM as HTML. |
| T-04.5-05  | Repudiation         | First-non-empty merge erases context | accept      | Operator chose D-Q1 = first-non-empty explicitly with rationale (stable framing); raw_payload retains every observed summary for audit.        |
| T-04.5-06  | Elevation of Privilege | Backfill script DB credentials       | accept      | Script reuses existing crawler session factory (same DB creds as production crawl path); no new credential surface introduced.                |
| T-04.5-07  | Denial of Service   | CBM decoder runs on every google_news insert/update | mitigate    | Pure-stdlib decoder, ~microsecond cost, no I/O. Bound input size implicitly (httpx response already bounded by feed).                          |

</threat_model>

<risk_section>

## What could break

1. **CBM decoder rotation (highest risk).** Google changes the protobuf wire format ~every 6-12 months (last known: Aug 2024). When it changes, ALL new Google News rows after that day get `resolved_url = NULL` and operator silently loses publisher URLs for new items. **Mitigation:** structlog warn on every decode failure is greppable in Cloud Run logs; operator can spot the spike. Existing rows are unaffected (Pass 2 of backfill already wrote what it could). The fallback (`url` still has the CBM token) means the system keeps working — degraded, not broken. ARC-001 still holds.

2. **Backfill running twice on production.** Both passes have `WHERE … IS NULL` guards, so the second run is a no-op by design. If operator panics and runs three times in a row, no damage. **Risk only if** the merge-strategy SQL is modified to drop the NULL guard — explicitly guarded against in T05 acceptance criteria.

3. **First-non-empty merge on topics with stale NULL descriptions.** A topic ingested before this phase has `description = NULL`. After T05 backfills it from raw_payload, a future Google News re-observation with a *different* summary will NOT overwrite (per D-Q1). This is the locked semantic — operator chose stability over freshness. If they change their mind, switch `COALESCE(topics.description, :new)` to just `:new` in `_bump_topic` (one-line change).

4. **HN topics show no description.** Expected per Fact 6 (HN feed has no summary). SPA gracefully omits the subtitle line when description is null. Operator might still ask "why no description on HN?" — answer is in SUMMARY.md "Known-NULL paths" section.

5. **NYT homepage RSS standfirst may be empty.** Not all NYT items carry `<description>`. Same graceful-NULL handling as HN. SUMMARY.md captures the empirical hit rate after the first post-deploy crawl.

6. **Alembic 0004 + ORM model drift.** If T02 adds the column to the migration but NOT to `core.models.TopicSource`, the next `insert_new` with `resolved_url=` kwarg will raise `TypeError`. T02 acceptance criteria explicitly requires both. Verified by `verify.schema-drift` query if needed.

7. **cloudbuild image tag confusion.** The tag is `$TAG_NAME` substitution; no hardcoded version in the build steps. The only "bump" work is updating doc-comments. The actual deploy command (`--substitutions=TAG_NAME=v0.5.0,...`) is operator-driven. SUMMARY.md states it explicitly.

8. **web/ build break from new optional types.** If `web/src/api/` has strict typing on `Topic` / `TopicSource` and `description` / `resolved_url` are not yet optional in the type, adding them in templates will fail TypeScript compile. T06 explicitly checks `web/src/api/` types and adds the optional fields if missing.

</risk_section>


```bash
docker compose exec api uv run --directory packages/core alembic downgrade -1
# or in Cloud Run: re-deploy v0.4.1 image which only knows about 0003.
# The extra resolved_url column is harmless to v0.4.1 code — it just ignores it.
```
Because 0004 is purely additive (nullable column, no constraint), rollback of the **app** image to v0.4.1 is sufficient; you do NOT need to also downgrade the migration. The orphan column stays in production until the next forward deploy.

**Backfill rollback:**
The backfill UPDATEs cannot be auto-reverted (we did not snapshot pre-backfill values — and `topic_sources.raw_payload['summary']` is the source of truth so we could re-derive). If the merge was wrong (e.g., we picked the wrong row per topic), TRUNCATE-and-re-derive:
```sql
UPDATE topics SET description = NULL WHERE id IN (<problem set>);
-- then re-run scripts/backfill_descriptions.py
```

**Decoder rollback:**
If the CBM decoder is misbehaving (returning nonsense URLs), one-line config-style disable: change `decode_google_news_url(item.url)` calls in T04 back to `None`. Existing decoded rows are not corrupted (they are just URLs); only NEW rows will stop populating resolved_url. SPA falls back to `url` automatically.
