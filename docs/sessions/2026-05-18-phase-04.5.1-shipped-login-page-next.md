## What Phase 4.5.1 actually fixed

**Bug discovered after v0.5.0 deploy:** Google News RSS `<description>` is
structurally an `<ol><li><a>` related-articles HTML fragment, not publisher
prose. ~29/30 prod topics had this HTML garbage backfilled into
`topics.description` by Plan 04.5-01 T05.

**Fix shape (operator choice m0141 = option A):** strip Google News
descriptions at the **parser level**, not via downstream HTML detection.
Clean, deterministic, no heuristic guesswork.

**Changes:**
- `RssSource.__init__` gained `capture_summary: bool = True` kwarg. When
  `False`, parser forces `RawItem.description = None` while preserving
  `raw_payload["summary"]` for forensic fidelity.
- `_RSS_SOURCES` in `services/crawler/src/crawler/app/composition.py` is now
  `list[tuple[str, str, bool]]`: `nyt_homepage=True`, `google_news=False`.
- New script `scripts/clear_google_news_descriptions.py` cleans existing dirty
  rows where every observed source is `google_news` (NOT EXISTS predicate
  preserves cross-source NYT-merged rows).

**Tests:** crawler suite `48 passed, 11 skipped`. The 11 skips are
**pre-existing** Phase 3 `run_once` signature drift in
`test_cross_source_dedup.py` â€” NOT caused by this plan.

**Locally verified:** dry-run 27 â†’ real 27 â†’ re-run 0 (idempotent âś…).
